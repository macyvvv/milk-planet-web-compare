# branch1: URL責務再設計案

## 結論
- URLの数を増やすか減らすかではなく、**1URL 1責務**に寄せる。
- 現状の問題は「ページ数」よりも「役割の混在」。
- proposal では、トップを入口に戻し、一覧・詳細・補助を分離する。

## 現時点の会話整理
- トップページの改修と `しすてむ&めにゅう` の改修は別項目として扱う。
- トップメニューは全ページ共通化の候補だが、トップページの表示崩れを起こさないことを優先する。
- トップカルーセルはトップ専用の初期化とし、他の `swiper-container` と干渉させない。
- `しすてむ&めにゅう` は `div#sys` の縦幅内に収め、上半分に見出し、下半分に各店リンクを配置する方向で検討する。
- キャスト一覧はオリジナルのフィルタ挙動を優先し、表示崩れを直した上で維持する。

## 責務の分け方

| URL種別 | 責務 | 現状の課題 | proposalの方針 |
|---|---|---|---|
| `index.html` | 入口 | 情報を詰め込みすぎ | ブランド概要と主要導線だけに限定 |
| `shop/index.html` | 店舗選択 | 各店舗情報が重い | 店舗比較と分岐の場に限定 |
| `shop/{shop}/index.html` | 店舗理解 | 店舗ごとの粒度が揃わない | その店舗の説明・写真・アクセス・SNSに限定 |
| `shop/{shop}/menu/index.html` | メニュー閲覧 | 詳細ページとの責務境界が弱い | 表示専用ページとして独立維持 |
| `cast/index.html` | キャスト一覧 | 店舗別アンカーが混在 | 全体一覧のハブに固定 |
| `recruit/index.html` | 採用情報 | 店舗ごとの導線が散る | 採用要件と応募導線に限定 |
| `planet-group/index.html` | グループ説明 | 入口と役割が近すぎる | グループ全体の文脈に限定 |
| `footer.html` | 共通部品 | ページとして扱いにくい | URL責務ではなく部品として扱う |

## 現状の整理で見える問題

### 1. トップの責務過多
- ブランド紹介
- 店舗導線
- キャスト導線
- システム導線
- イベント導線
- SNS導線
- 採用導線

これらを1ページに載せると、入口としては強いが、比較検討の視点では重すぎる。

### 2. 店舗一覧の責務が曖昧
- 一覧ページであるはずなのに、店舗個別の見せ方を抱え込んでいる
- 一覧と詳細の境界が弱く、UIの一貫性が崩れる

### 3. メニューURLの位置づけが弱い
- `menu/index.html` は表示専用としては妥当
- ただし、店舗詳細と近すぎると「どちらを見ればよいか」が曖昧になる

### 4. 埋め込みや外部導線が本体に侵食している
- Instagram、TikTok、地図、LINE、X、メールなどが混在する
- これはユーザー行動としては重要だが、ページ責務としては別枠で扱うべき

## proposal のURL方針

1. 入口は `index.html` に集約する
2. 比較は `shop/index.html` で行う
3. 理解は `shop/{shop}/index.html` で行う
4. 表示補助は `menu/index.html` に分離する
5. 外部コミュニケーションはSNSや採用ページに逃がす
6. **DOMの表示/非表示制御による「疑似SPA」は、単一データのフィルタリング（キャスト一覧等）にのみ限定する。店舗やシステムなど責務が異なる情報の切り替えには用いず、物理的なURL遷移を伴うMPAとする。**

## 例外扱い
- ページ内アンカーは URL ではなく同一文書内操作として扱う
- `footer.html` はページではなく部品として扱う
- `eventlist.js` の動的生成は URL 設計ではなく描画設計として扱う

## この案の狙い
- プレゼンで「新旧比較」がしやすくなる
- 更新作業を増やしすぎない
- 将来の再利用時にも責務が壊れにくい
- 既存素材を使い回しながら、見せ方だけ改善しやすい

## PC版の見た目に関する追加方針

### 1. トップのフローティングメニューは常時露出しない
- 現状の固定表示は、他要素と重なると可読性が落ちる
- PCでは特に、ナビがコンテンツに干渉しやすい
- そのため、**ハンバーガーメニューにまとめて必要時のみ開く方式**に寄せる
- ただし、SP同様の隠しすぎにはせず、PCでは右上固定のトリガーを残す
- 表示内容は主要導線に絞る
  - ほぉむ
  - わくせい
  - きゃすと
  - しすてむ&めにゅう
  - いべんと
  - りくるうと

### 2. トップのカルーセルはファーストビューを占有しすぎない
- 現状は高さが大きく、最初にブランド全体像より画像だけが先に来る
- そのため、**画面縦の半分程度までに収める**のが妥当
- 画像は維持しつつ、見せ方を圧縮する
- 目安は次の通り
  - カルーセル高さを viewport の 40〜50% 程度に制限
  - `object-fit: cover` で見切れを許容しつつ崩れを防ぐ
  - カルーセル直下にコンセプト文と主要CTAを早めに置く
- トップ自体の構成は維持してよい
- 変更対象はカルーセルのサイズと比率に限定する

### 3. 解釈
- これは単なる見た目調整ではなく、**情報の優先順位修正**
- ナビは常時見せる必要がない
- カルーセルは主役ではなく、入口の印象づけに留める
- その結果、PC版で「何のサイトか」「どこへ進めばよいか」が早く理解できる
